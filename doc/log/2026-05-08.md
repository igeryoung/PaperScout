**Deviations from plan:**

- **`src/server/schema/*.ts` deliberately do NOT use `import 'server-only'`.** They are pure zod (no DB, no env) and must be loadable by the validate CLIs running under plain `tsx`. The convention "everything in `src/server/` is server-only" is amended: the schema submodule is exempt because it contains only pure data definitions.
- **`paper_duplicates` recording is conservative.** When the matcher fires on `FUZZY_TITLE` or `NORMALIZED_TITLE` we currently record a self-link (canonical=duplicate=match.paperId) rather than introducing a separate "duplicate paper row." V1 doesn't need to model duplicate papers as distinct entities; only the audit trail. Revisit if PRD §22 metrics need a real duplicate count.
- **`fastest-levenshtein` chosen over `fast-levenshtein`** — newer, faster, MIT-licensed. Plan said simply "fuzzy match"; no other lock-in.

**Surprises / learnings:**

- The `server-only` package literally throws when imported in Node. Vitest needs the alias-to-stub workaround; tsx-run scripts need to avoid the import altogether. Adding `'server-only'` to truly-pure modules creates more friction than safety — the "is this server-only" question deserves per-file judgment, not a blanket convention.
- Prisma's `upsert` with composite unique that includes a nullable column is a typecheck dead-end (the where-helper rejects null). The pragmatic fix is `findFirst + (update | create)`. Documented in the feedback repo.
- Sample data revealed a schema modeling decision: `evaluationStage = ABSTRACT_SCREENING` should imply `pdfAnalysisStatus = null` (not "UNAVAILABLE"). UNAVAILABLE belongs to FULL_PDF rows where the PDF was attempted but unobtainable. Encoded as a zod refinement.
