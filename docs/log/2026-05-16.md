# 2026-05-16 — Bilingual evaluations (i18n preparation, full pipeline)

## What changed

Shipped end-to-end bilingual support (en + zh-TW) for evaluation narrative — from the LLM skill output, through the on-disk JSON, the zod schema, the Postgres columns, the ingest path, and the read-only UI (with a locale switcher in the header).

Full per-area detail is in [`doc/STATE.md`](../STATE.md) under "Phase 4 extension — Bilingual evaluations (2026-05-16)". Catalog of the data contract is in [`doc/data-contract.md`](../data-contract.md). Architecture rules are in [`doc/ARCHITECTURE.md`](../ARCHITECTURE.md) under "Internationalization (i18n)".

## Why

Preparing the system for i18n. The home page chrome was already hardcoded zh-TW while the rest of the UI was English; the goal was to flip a single switch and have both the static chrome and the LLM-generated narrative render in the user's chosen locale.

## Decisions taken

1. **Per-field locale map, not a top-level `localized` block, not suffix columns.** Each translatable field becomes `{ en, "zh-TW" }` (lists become `{ en: string[], "zh-TW": string[] }`). Pros: easy field-level diffs, trivial to add a third locale, queryable as `summary->>'en'` in SQL. Cons: drops column-level NOT NULL/length checks → moved to zod's `superRefine`.

2. **JSONB columns, not parallel `_zh_tw` columns or a monolithic `localized JSONB` blob.** Less migration churn, no doubling of column count, no destructive drops.

3. **Cookie-based locale, not URL path prefixes.** No `/[locale]/...` restructure, no middleware. `?locale=` URL override works for the page body (used for sharing). The `<html lang>` attribute only honors the cookie — easy follow-up if we want them in sync.

4. **`DEFAULT_LOCALE = 'zh-TW'`** to match the existing home-page chrome (`為你推薦`, `熱門趨勢`, etc.) that this project shipped with.

5. **Tags stay English-only.** They're a lowercase keyword search/filter index, not user-facing prose. Translating fragments would fragment the index and confuse `/library?tags=...` deep links.

6. **`figure.caption.en` is verbatim from the PDF (≤ 240 chars); `caption['zh-TW']` is a faithful translation.** Keeps the citation-style guarantee on English while still letting Chinese-locale users read the caption.

7. **Cross-locale fallback in `pickLocalized`.** Legacy DB rows post-migration have `zh-TW: NULL`; we fall back to the populated locale rather than rendering empty. Important during the rollout window before all evaluations are re-run.

## Non-destructive migration

Migration `20260516105100_bilingual_text` hand-edits the `ALTER COLUMN ... TYPE JSONB` statements with `USING jsonb_build_object('en', col, 'zh-TW', NULL)` clauses so existing dev rows are preserved as `{ en: <original>, 'zh-TW': null }`. `strengths` and `weaknesses` (already JSONB) are wrapped in-place with the same pattern. Applied to both `public` and `paperscout_test` schemas.

## Follow-ups not done in this change

- `scripts/prompt-eval/reference/normalized/F{1..5}.json` were NOT regenerated — they remain frozen on the old shape until the harness is re-run end-to-end. The raw fixture at `scripts/prompt-eval/reference/raw/2026-05-10-2142/evaluations.json` was wrapped by a one-shot transform (en duplicated into zh-TW) so the integration test that ingests it still passes; this is a placeholder until a real bilingual harness run happens.
- Pre-bilingual `data/runs/*/evaluations.json` files in the repo (e.g. `2026-05-08-1743`, `2026-05-12-1014`) will fail the new zod schema on re-ingest. Re-evaluate them with the updated skill if you need to re-ingest.
- `<html lang>` doesn't honor `?locale=` — only the cookie. Plumb searchParams into `RootLayout` (or a tiny middleware) when this matters.
