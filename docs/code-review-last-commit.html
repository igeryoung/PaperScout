Executive Summary

No blocking security or correctness issue was found. The largest risks are robustness gaps in the newly exposed mutation endpoints and a race in first-time default collection creation. The commit builds and passes existing automated checks, but the new library API/repository paths do not appear to have dedicated tests.

Findings

MEDIUM Invalid or missing paper IDs can bubble up as server errors

src/app/api/library/papers/[paperId]/route.ts:55,91,124; src/server/repos/library.ts:237-278

The paper mutation route trusts the dynamic paperId segment and passes it directly into Prisma upserts/deletes. The body validates collectionId as a UUID, but the path parameter is not validated and the repository does not check that the paper exists before creating user_papers or paper_collection_items. A malformed UUID or deleted paper can therefore produce Prisma/foreign-key errors and return a 500 instead of a controlled 400/404 response.

Recommendation: validate paperId with the same UUID schema used for collectionId, check paper existence before upsert, and map Prisma errors such as invalid UUID or foreign key failures to explicit client responses.

MEDIUM Default collection creation is race-prone

src/server/repos/library.ts:35-54

ensureDefaultCollection performs a read, then a second read by name, then a create. Two concurrent first requests for the same user can both observe no default/name match and race into paperCollection.create. Because the table has @@unique([userId, name]), one request can fail with a unique constraint error, which can break initial library page loads or first-time collection mutations.

Recommendation: make this path idempotent with an upsert on userId_name, or catch the unique-constraint failure and re-read the collection. If exactly one default collection per user is required, also consider a database-level uniqueness guard for is_default.

LOW English users receive a Chinese default list name

src/server/repos/library.ts:33

The default collection name is hard-coded as a Chinese string in the server repository. Since the library page supports en and zh-TW, a first-time English user will see a localized interface with a non-localized default list name.

Recommendation: use a locale-neutral default such as Default, or store a stable system collection flag and render the display label through i18n instead of persisting a localized label.

LOW Sign-in required copy bypasses i18n

src/app/library/page.tsx:51-66

The signed-out library state uses hard-coded English text. Users in the zh-TW locale get localized navigation and library labels elsewhere, but this new entry state does not use the existing message catalog.

Recommendation: add the signed-out title, body, and button label to src/i18n/en.ts and src/i18n/zh-TW.ts, and pass the localized copy into SignInRequired.
