結論

Phase 2 的主幹方向正確：三個來源已正規化成同一個 Candidate schema，aggregator 有 quota / dedup，API route 也改用 Next.js 16 的 after()。本報告原本建議 REQUEST CHANGES；2026-05-10 已依決策更新為「允許 partial completion」，並補上 structured logging、10 分鐘 timeout、 source identity persistence、integration opt-in gating 與文件原則。

修正更新

  • Partial collection 現在是明確合約：不足 30 筆仍可 COMPLETED，log 會標示 isPartial=true。
  • Source fetch 使用共用 timeout helper，預設 10 分鐘；/api/runs route 設定 maxDuration = 600。
  • PaperSource existing check 改為 source identity 層級，不再只看 paperId + source。
  • Phase 2 runtime log 改用 src/lib/logger.ts structured events；原則寫入 plan/IMPLEMENTATION_PRINCIPLES.md。
  • npm run test:integration 改為需要 RUN_INTEGRATION=1 與 DATABASE_URL_TEST 才執行，避免 stale local DB URL 造成例行驗證失敗。

主要問題

來源不足時仍會把 run 標記為完成

HIGH

collectFromAllSources() 在所有 source 回傳數量不足時會直接回傳短列表， 測試也鎖定這個行為：tests/unit/sources/index.test.ts:164 到 172 期望只回傳 9 筆。接著 runCollectionInBackground() 不檢查 candidate 數量，persist 完後直接在 src/server/pipeline/runner.ts:36 設為 COMPLETED。

風險：如果三個 API 暫時都掛掉、HF 被 CV heuristic 過濾太多、OpenReview 回傳很少，run 仍會顯示成功，但 DB 可能只有 0 到 29 筆結果。使用者與後續 Phase 3 會把不完整資料當成成功輸入。

狀態：已採用「允許 partial」的合約，並更新 Phase 2 README。runner 會完成已持久化的資料， structured log 會記錄 targetCount、collectedCount 與 isPartial。

PaperSource existence check 太粗，會吞掉同來源的新 provenance

MEDIUM

persistCandidates() 在 existing paper path 使用 sourcesRepo.exists(paperId, source) 判斷是否新增來源： src/server/pipeline/persist.ts:97 與 107。但 repo 的實作只看 paperId + source，見 src/server/repos/sources.ts:21 到 22。

風險：同一篇 canonical paper 若之後被同一 source 的不同 URL / slug / sourcePaperId 指到，現在會因為已有該 source 而不寫入新的 paper_sources。schema 本身只限制 (source, sourcePaperId) 唯一，並沒有禁止同一 paper 有多個同來源記錄，因此目前判斷比 DB model 更窄，會造成來源追蹤資料遺失。

狀態：已改成 existsIdentity(paperId, source, sourcePaperId, sourceUrl)， 用 sourcePaperId 或 sourceUrl 區分 provenance identity。

外部 fetch 沒有 timeout，背景任務可能卡到平台上限

MEDIUM

三個 source client 都直接呼叫 injected/global fetch： src/server/sources/arxiv.ts:99 到 106、 src/server/sources/openreview.ts:94 到 101、 src/server/sources/huggingface.ts:129 到 136。沒有 AbortController timeout，也沒有每來源的 latency budget。

風險：after() 是正確 primitive，但 Next 文件明確說它仍受 route/platform max duration 限制。任一來源連線卡住時，整個 Promise.allSettled() 會等它結束，run 可能長時間維持 RUNNING，最後被平台中止但 DB 狀態未必被標為 failed。

狀態：已建立共用 fetch helper。依本次決策，timeout 設為 10 分鐘；timeout 後該 source reject，aggregator 會按現有設計 log error 並使用其他來源補量。

Integration test 環境不可用時不是乾淨 skip

MEDIUM

README close-out 說 npm run test:integration 在未設定 DATABASE_URL_TEST 時會跳過。但實際執行時載入了本機 env，嘗試連 localhost:5435，DB 不在時整個 suite fail。相關 gating 在 tests/integration/setup.ts:8 到 14，test 使用 describe.skipIf(!SHOULD_RUN_INTEGRATION)，見 tests/integration/collect-persist.test.ts:58。

風險：CI 或新開發機只要有 stale DATABASE_URL_TEST，但測試 DB 未啟動，就會失敗； 這不是壞事，但與文件「skips silently if unset」容易造成誤判。更大的問題是沒有快速 preflight 說明缺哪個服務。

狀態：已採用 RUN_INTEGRATION=1 顯式開關。沒有 opt in 時 integration suite 會乾淨 skip；有 opt in 但 DB 不可用時才 fail。

arXiv rate limiter 已被文件列為風險，但實作沒有接上

LOW

fetchArxiv() 是單次 fetch，沒有 3 秒 limiter。Phase 2 close-out 說 「one-shot per-run flow」暫時不需要，但 README risks 仍把 arXiv 429 列為 mandatory pitfall。若之後加重試、排程、或多使用者同時按 POST /api/runs，這個缺口會被放大。

建議：至少把 rate limit helper 放在 arXiv client 邊界，並讓未來 loop/retry 無法繞過它；或在 README 明確改成「目前單次 request 不節流，retry/loop 前必須補上」。

正向觀察

  • POST /api/runs 使用 after()，符合 Next.js 16 route handler 的背景 callback 模式。
  • Source parser 都會用 CandidateSchema.safeParse() 擋掉不合 schema 的資料，這讓 DB 寫入前的資料形狀比較穩。
  • Aggregator 的 within-batch dedup 會合併 additionalSources 與 codeUrls，也已有測試覆蓋 arXiv/HF collision。
  • npm test 已通過：10 個 test files、58 個 tests。

驗證結果

  • PASS npm test：10 passed / 58 passed。
  • SKIP npm run test:integration：未設定 RUN_INTEGRATION=1 時乾淨跳過，1 file / 1 test skipped。

建議修正順序

  1. 先決定 Phase 2 run 是否允許 partial；若不允許，runner 必須在少於 30 筆時標為 failed。
  2. 替 source fetch 加 timeout，避免背景任務卡住。
  3. 把 sourcesRepo.exists() 改成 source identity 層級，避免吞 provenance。
  4. 讓 integration test 的啟動條件與錯誤訊息清楚可預期。
  5. 補上或明確文件化 arXiv rate limiting 的觸發條件。
