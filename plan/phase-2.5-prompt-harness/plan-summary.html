Phase 2.5 Prompt Harness 計畫摘要

本階段要在進入 Phase 3 排名管線前，先用固定論文 fixtures 驗證 Stage-1 摘要篩選與 Stage-2 PDF 深讀 prompt，確保輸出符合既有 EvaluationRecord contract，且分數排序符合 reviewer 直覺。

Offline Harness Skill-driven No DB / No UI
5 手選 fixture papers，覆蓋強弱與偏誤案例
2 評估階段：abstract screening 與 full-PDF analysis
1 唯一輸出 contract：既有 EvaluationRecord[]
0 不新增 Anthropic SDK、不要求 app 內 API key

High Level

為什麼需要 Phase 2.5

Phase 2 已經完成 source collection，Phase 3 會把 ranking 接進 pipeline。若 prompt 品質不穩，Phase 3 debug 會混在 LLM 判斷、 schema、pipeline 與 DB persistence 之間。Phase 2.5 先把 prompt 行為固定下來，降低後續整合風險。

目前架構決策

這個 repo 目前不在 app code 裡直接呼叫 Anthropic，也不要求 ANTHROPIC_API_KEY。Prompt 執行仍由 /evaluate-papers skill 負責；harness 只負責建立 fixture run、驗證 JSON、檢查分數與保存 regression output。

預期目標

Schema 穩定

Skill 輸出的 evaluations.json 必須通過 EvaluationsFileSchema，並且每筆 joinKey 都能對應到 fixture candidate。

分數符合直覺

至少 4/5 fixtures 通過 soft bounds；F2 novelty 應偏低，F5 experimentalQuality 應偏低，F4 不應因 unknown authorship 被過度壓分。

可回歸

接受的 raw outputs 與 normalized snapshots 會保存到 scripts/prompt-eval/reference/，供 Phase 3 和未來 prompt 調整時比較。

主要內容

區塊 內容 產物
Fixtures 建立 F1-F5 五類論文案例：強 novelty、incremental tweak、名校弱實驗、 unknown group 強方法、hype-driven 弱 claims。 metadata.json、bounds.json、 optional paper.pdf
Prompt Source 以 .claude/skills/evaluate-papers/SKILL.md 為主要 prompt source；若 .agents/skills/ 仍使用，需保持同步。 更新後的 skill prompt
Fixture Builder 把 fixtures 轉成一個 5-entry CandidateRecord[] run，並驗證 candidates.json shape。 scripts/prompt-eval/build-fixture-run.ts
Evaluation Checker 驗證 evaluations.json、檢查 joinKey、重算 total、 套用 bounds、輸出排名與錯誤報告。 scripts/prompt-eval/check-evaluations.ts
Regression Snapshot 保存 raw skill output，並產生 normalized copy 供後續比較。 scripts/prompt-eval/reference/

執行流程

1 建立 fixtures 撰寫 F1-F5 的 metadata、bounds，並準備本地 PDF。
2 產生 fixture run 執行 npm run prompt:fixtures，輸出 candidates.json。
3 執行 skill 對 fixture run 呼叫 /evaluate-papers。
4 Schema 驗證 執行 npm run validate:evaluations。
5 Bounds 檢查 執行 npm run prompt:check -- <run-dir>。
6 保存回歸資料 通過後保存 raw 與 normalized reference outputs。

驗收標準

必須通過

  • prompt:fixtures 產生有效的 5-candidate fixture run。
  • /evaluate-papers 能產生 evaluations.json。
  • validate:evaluations 通過。
  • prompt:check 通過，且至少 4/5 soft bounds pass。
  • reference outputs 已保存。

品質判斷

  • 整體排名大致符合 F1 > F4 > F3 > F5 > F2。
  • F2 novelty 明顯偏低。
  • F5 experimentalQuality 明顯偏低。
  • F4 不因 unknown authorship 被系統性低估。
  • PDF analysis 有引用 tables、figures 或 ablations 的證據。

明確不做

不新增 direct LLM client

本階段不建立 src/server/llm/client.ts、 src/server/llm/pdf.ts、src/server/llm/schema.ts 或 src/server/llm/prompts/*。除非另行決定 Phase 3 要改成 app code 直接呼叫 Anthropic。

不碰 DB / UI

Harness 不寫入 Postgres、不修改 library page、不跑完整 pipeline。 它只建立 fixtures、驗證 skill output，並把接受的結果保存成 regression data。

主要風險

風險 處理方式
Skill invocation 是手動流程 這是目前架構的刻意選擇；harness 專注驗證輸出，不負責呼叫 LLM。
兩份 skill prompt 漂移 若 .claude/skills/ 與 .agents/skills/ 都保留，Phase close 前要同步，或指定一份為 canonical。
LLM total 加總錯誤 Checker 報告 mismatch；只在 normalized copy 修正，不靜默修改 raw output。
PDF URL 失效 預設使用本地 paper.pdf，URL 只用於刷新 fixtures。
Unknown authors 被低估 F4 專門作為 regression check，失敗時收緊 prompt。
