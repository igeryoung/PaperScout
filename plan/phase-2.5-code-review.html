審查發現

High Prompt checker 沒有檢查 5 個 fixture 是否全部完成評估

位置： scripts/prompt-eval/check-evaluations.ts:110 到 scripts/prompt-eval/check-evaluations.ts:123

目前 gate 只把 schema-valid 筆數拿來和 summary.records.length 比較，並接受 boundsPassedCount >= 4。它沒有驗證 manifest 裡的每個 fixture 都剛好產生一筆 matched evaluation。把 F5 從 run 裡移除後， 指令仍會 exit 0，因為剩下四筆都 schema-valid，而且四筆 bounds 都通過。

這會削弱 Phase 2.5 的核心合約。Harness 可能 freeze 或 approve 一個靜默跳過 fixture 的 prompt run，包含原本明確用來檢查 hype / weak-claims 的 regression case。

修正狀態：已從 manifest 推導 expected fixture ids，當任何 fixture missing、duplicated 或 unmatched 時直接 fail。也已補 unit test 覆蓋 missing 和 duplicate evaluation。

Medium Normalization 可能留下 stale reference files

位置： scripts/prompt-eval/normalize-evaluations.ts:95 到 scripts/prompt-eval/normalize-evaluations.ts:115

Normalizer 會為每筆 matched evaluation 寫出一個檔案，但沒有檢查完整 fixture set 是否都已寫出，也沒有在寫入前清掉既有的 reference/normalized/F*.json。如果後續 run 漏掉 F5，舊的 F5.json 仍可能留在目錄裡，讓 reference directory 看起來是完整的。

修正狀態：Normalizer 已和 checker 共用 manifest completeness check，遇到 missing、duplicate 或 unexpected fixture 時 fail。寫入前也會移除 本次 manifest 對應的舊 normalized outputs，避免 stale file 留在目錄裡。

Low Generated next-env.d.ts 被手動變更，不應納入本 phase

位置： next-env.d.ts:3

這個檔案本身標註不應手動編輯；本機 Next 16 文件也說 next-env.d.ts 會由 next dev、next build 或 next typegen 產生，並建議加入 gitignore。這次 Phase 2.5 diff 把 route type import 從 .next/types 改成 .next/dev/types，這和 prompt harness 無關，而且會依最後執行的 Next command 產生 churn。

修正狀態：Phase 2.5 變更已 revert 這個 generated-file delta。 是否要依照目前 Next 文件停止追蹤 next-env.d.ts，可另開 cleanup 處理。
